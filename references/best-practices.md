# 邮件处理最佳实践 v2.0

## 1. 邮件分类标准（7级）

### 🔴 安全告警 (security_alert)

最高优先级，满足以下任一条件：
- API Key / Secret / Token 泄露（GitHub、GitGuardian等平台通知）
- 账户异常登录或被盗用
- 云服务资源被挂起/停用（Zilliz、AWS、Azure等）
- 代码仓库中检测到敏感信息
- SSL证书过期、域名到期

**处理方式**：立即标注并通知用户，建议轮换密钥或检查账户。

### 🟠 紧急 (urgent)

满足以下任一条件：
- 发件人为 VIP 列表中的联系人
- 邮件正文中包含明确截止日期且在 3 天内
- 含有"紧急"、"URGENT"、"ASAP"、"请立即"、"马上"等催促词
- 邮件标记为"高重要性"
- 系统/运维告警邮件（含 ERROR、ALERT、故障等关键词）

### 🟡 待回复 (needs_reply)

满足以下任一条件：
- 邮件中包含问号（?或？）
- 含有"请确认"、"请回复"、"请告知"、"是否可以"等请求
- 需要做决策或选择
- 会议邀请中需要回复接受/拒绝
- 客户/合作方发来的询问

### 🟢 仅阅读 (info_only)

满足以下任一条件：
- CC 抄送的邮件（非直接收件人）
- 系统通知、订阅内容、周报/日报
- 无需回复的公告/通知
- 已有其他人回复的讨论线程
- OAuth 应用授权通知（确认是本人操作即可）

### 🔘 验证码 (verification)

满足以下条件：
- 邮件主题或正文包含"验证码"、"verification code"等
- 包含6位数字验证码
- 已过期（验证码有效期通常5-15分钟）

**处理方式**：统一跳过，不生成摘要和回复，标注"已过期/可忽略"。

### 📰 自发送 (self_sent)

满足以下条件：
- 发件人地址 == 收件人地址
- 定时任务推送（每日新闻、报告等）
- 自动化脚本产生的通知

**处理方式**：仅摘要，不回复。

### ⚫ 垃圾邮件 (spam)

满足以下任一条件：
- 含有大量退订链接
- 可疑发件人地址（随机字符串+免费邮箱）
- 推广/营销内容为主（会员续费、折扣、积分兑换）
- 标记为垃圾邮件的系统识别结果

**处理方式**：自动过滤，不生成摘要。

## 2. 紧急程度判断规则

### 多维度评分模型

```
总分 = 安全告警加分 + 发件人权重 + 关键词权重 + 时间权重 + 重要性标记
```

| 维度 | 规则 | 分值 |
|------|------|------|
| 安全 | API Key泄露/账户异常 | +100（直接升级为最高） |
| 发件人 | VIP 列表 | +30 |
| 发件人 | 直属上级/客户 | +20 |
| 发件人 | 同事 | +10 |
| 发件人 | 系统/通知 | +5 |
| 关键词 | "紧急/URGENT/ASAP" | +25 |
| 关键词 | "deadline/截止/到期" | +20 |
| 关键词 | "请/需要/务必" | +10 |
| 时间 | 3天内截止 | +25 |
| 时间 | 7天内截止 | +15 |
| 时间 | 无明确时间 | +0 |
| 标记 | 高重要性 | +20 |

**分级阈值**：
- 安全告警命中 → 🔴 安全告警
- ≥ 50 分 → 🟠 紧急
- 25-49 分 → 🟡 待回复
- 10-24 分 → 🟢 仅阅读
- < 10 分或命中 spam 规则 → ⚫ 垃圾邮件

## 3. 安全告警处理规范

### 常见安全告警类型

| 类型 | 来源 | 严重程度 | 建议操作 |
|------|------|----------|----------|
| API Key 泄露 | GitHub/GitGuardian | 极高 | 立即轮换密钥，检查代码历史 |
| 账户异常登录 | Microsoft/Google | 高 | 修改密码，启用2FA |
| 资源挂起 | Zilliz/AWS/Azure | 中 | 决定是否重新激活 |
| OAuth 授权 | GitHub | 低 | 确认是本人操作即可 |
| SSH Key 添加 | GitHub | 低 | 确认是本人操作即可 |

### 脱敏要求

安全告警中的敏感信息（如具体的 API Key 值、Token）在报告中必须脱敏：
- API Key：只显示前4位和后4位，中间用 `****` 替代
- Token：只显示类型（如 `ghp_****`），不显示完整值

## 4. 回复风格指南

### 正式版 (formal)

适用场景：对外客户、上级领导、合作方、首次沟通

- 开头："您好 [称呼]，"
- 正文：完整句子，避免缩写
- 结尾："此致敬礼" 或 "顺祝商祺"
- 称呼：使用职位/姓氏，不用昵称

### 简洁版 (concise)

适用场景：同事、内部沟通、已熟悉的合作方

- 开头："Hi [称呼]，" 或 "[称呼]，"
- 正文：关键词、短句、直接回答问题
- 结尾：简单致谢即可
- 可以用列表/要点形式

### 追问版 (follow-up)

适用场景：信息不足、需要澄清、需要更多细节

- 用"想确认一下…"、"请问…"开头
- 列出具体需要确认的问题点（编号列表）
- 表达合作意愿
- 限定回复期望时间

## 5. 隐私与安全规范

### 脱敏规则

以下信息在摘要中必须脱敏处理：

| 信息类型 | 脱敏方式 | 示例 |
|----------|----------|------|
| 身份证号 | 保留前3后4 | 110***1234 |
| 银行卡号 | 保留后4位 | ****5678 |
| 手机号 | 保留前3后4 | 138****1234 |
| 薪资数字 | 用"[已脱敏]"替代 | 月薪[已脱敏] |
| 合同金额 | 保留数量级 | 约X万级别 |
| 密码/Token | 完全隐藏 | [已隐藏] |
| API Key | 显示前4后4 | ghp_****abcd |

### 发送安全

- 回复草稿必须经用户确认后才能发送
- 不自动转发含敏感信息的邮件
- 不在摘要中包含邮件附件内容（除非用户明确要求）
- 不将邮件内容发送到非指定的第三方服务

### 日志安全

- 脚本运行日志中不记录邮件正文
- IMAP 连接信息在脚本退出后清除
- 不将邮件内容写入持久化文件（除非用户明确指定）

## 6. 中文日期处理规范

### 绝对日期

直接解析为标准日期：
- `2026-05-28` → 2026年5月28日
- `5月28日` → 当前年的5月28日
- `28/05/2026` → 2026年5月28日

### 相对日期

基于当前日期计算：
- `今天/今日` → 当天
- `明天/明日` → 当天+1天
- `后天` → 当天+2天
- `下周一` → 下一个周一
- `本周末` → 本周六
- `月底前` → 本月最后一天
- `下月初` → 下月1日

### 模糊日期

无法精确计算时保留原始文本：
- `尽快/尽快处理` → [尽快]
- `等通知` → [待通知]
- `适时` → [适时]
